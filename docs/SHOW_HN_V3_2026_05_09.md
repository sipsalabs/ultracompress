## Post body (~400 words)

UltraCompress is a 5-bit transformer compression format with a property the 4-bit alternatives don't have: **the compressed weights reconstruct bit-identically to a SHA-256 reference** taken at pack write time. AWQ, GPTQ, HQQ, and bitsandbytes-NF4 all introduce small run-to-run drift between the trainer's quantized eval and the customer's deployed inference. UltraCompress doesn't — `uc verify` re-checks the SHA-256 over the reconstructed tensor bytes on the consumer side.

Mechanism, in two sentences. The pack stores a per-row k-means scalar grid (5 bpw codes), per-block fp32 absmax scales (block_size=64), and a low-rank low-rank correction `(U, V, alpha)` trained per layer against teacher hidden-state KL. Reconstruction at inference is closed-form: `W = scalar_dequantize(codes) * absmax + alpha * low_rank(U, V)`, which is the same expression the trainer used during eval — hence the bit-identity.

Numbers from the validation matrix (FineWeb-edu held-out tail, no calibration overlap, RTX 5090, full table at `docs/BENCHMARKS_2026_05_08.json`):

- `Qwen3-1.7B-Base` (qwen3 dense): PPL ratio **1.00401x**, n=30, seq_len=1024
- `Yi-1.5-9B` (llama dense): PPL ratio **1.00414x**, n=50, seq_len=1024 — tightest >8B-param 5-bit ratio we've measured
- `Mamba-2.8B` (state-space, not transformer): PPL ratio **1.0119x**, scalar-only — first lossless 5-bit SSM compression artifact we know of
- `Hermes-3-Llama-3.1-405B`: compressed end-to-end on a single 32 GB consumer GPU via per-layer streaming. Pack complete at ~251 GB. Full-run PPL eval is in flight on `cuda:1`; partial-run during compression gave 1.0071x. HF artifact uploading.

22 architectures validated end-to-end so far (Qwen3, Llama-3.1, Mistral, Mixtral, Phi-3 / Phi-3.5-MoE, SmolLM2, OLMo-2, Mamba, plus base/instruct pairs). 13 of them sit under 1.013x.

Reproduce in three commands:

```
pip install -U ultracompress
hf download SipsaLabs/qwen3-1.7b-base-uc-v3-bpw5 --local-dir ./qwen3-1.7b-base
uc verify ./qwen3-1.7b-base
```

Expected output: `VERIFY: PASS — pack format integrity confirmed; lossless reconstruction guaranteed.`

What we are not claiming: faster inference than AWQ/GPTQ at the kernel level (we re-use PyTorch matmul, no custom CUDA yet); lossless below 5 bpw (sub-3 bpw still hits the documented Qwen3-fragility wall); per-architecture downstream-task validation (PPL only). 13 honest negative results from this week are catalogued in `docs/HONEST_NEGATIVE_RESULTS_2026_05_08.md`, including a refuted `k_proj`@6bpw cure on Qwen3-1.7B.

## First comment from OP (~300 words, paste as the first reply within 30 seconds of submission)

OP here. Anticipating the obvious question — "you're calling 5-bit compression *lossless* but the PPL ratio is 1.004x, not 1.000x, so what gives?" — let me preempt that.

"Lossless" here means a specific, narrow, mathematically checkable property: **the weights the customer reconstructs from the pack are bit-identical, byte-for-byte, to the weights the trainer evaluated during compression.** SHA-256 over the reconstructed tensor bytes matches at pack write time and `uc verify` re-checks it on the consumer side. That's it. Pack format is documented at `docs/HF_MODEL_CARD_HERMES_3_405B.md` if you want the closed-form reconstruction expression.

What that property gets you that AWQ / GPTQ / HQQ / bitsandbytes-NF4 don't: **zero run-to-run drift between the trainer's quantized eval number and the customer's deployed inference number.** Those formats have small but real reproducibility gaps — different CUDA versions, different `torch_dtype` defaults, different per-channel scale rounding paths produce slightly different `Wq` at the customer's machine than at the trainer's. UltraCompress eliminates that class of bug because the reconstruction formula is closed-form over stored fp32 metadata, not a dequant kernel with implementation freedom. For regulated-industry deploys (defense, finance, medical) where the compliance question is "does the production model behave bit-exactly the same as the eval model the auditor signed off on," this matters. Full landscape comparison: `docs/COMPETITIVE_LANDSCAPE_v3_LOSSLESS_2026_05_08.md`.

What we're **not** claiming: lossless against the bf16 base model. The 1.004x PPL ratio is real and is the lossy step (5-bit scalar quantization + low-rank correction). The lossless property is between the trainer's compressed weights and the customer's reconstructed weights, not between the compressed model and the original.

Open issues, on the record: the `k_proj` outlier cure (promote `k_proj` to 6 bpw on a 5-bpw base) was **refuted** apples-to-apples on Qwen3-1.7B-Base last night — see negative-result entry §12 in `docs/HONEST_NEGATIVE_RESULTS_2026_05_08.md`. V4 cure direction (rank redistribution at constant total budget, not per-Linear bpw redistribution) is in flight, no number to share yet. 13 documented refutations from this week are in the same file. We catalogue failures for the same reason we catalogue wins.

Scrutiny welcome. The pack format is fully documented in the repo, the verification script runs in seconds on consumer hardware, and the JSON benchmark file links every claim back to a per-eval JSON with seed and host metadata. If something doesn't reproduce, file an issue — that's the most useful comment you could leave.
